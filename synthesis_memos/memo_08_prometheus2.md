# Synthesis Memo 08 — Prometheus 2

**Paper**: Kim et al., "Prometheus 2: An Open-Source Language Model Specialized in Evaluating Other Language Models," 2024  
**Date**: 2026-04-29  
**Author**: Kemeriya  
**Status**: Path B reading — reference architecture for a small domain-specific judge model

---

## Core Claims

Prometheus 2 is a 7B and 13B parameter judge model trained on a large feedback collection (Feedback Collection, 100K+ instances) generated by GPT-4. It achieves competitive performance with GPT-4-as-judge on Vicuna and MT-Bench evaluation tasks using a structured evaluation format: the judge receives a score rubric, a reference answer, and the candidate output, then produces a score and a feedback explanation.

Key design choices: (1) separate model variants for absolute scoring (pointwise) and relative comparison (pairwise); (2) training on feedback explanations, not just scores — the model learns to reason about quality before outputting a number; (3) fine-grained rubric conditioning, where the rubric is part of the prompt, allowing the same model to evaluate against different criteria.

---

## Where I Agree

The fine-grained rubric conditioning is the right design principle and is applied in scoring_evaluator.py's LLM judge path: each tone marker prompt specifies the exact definition of the marker ("Clear, brief, actionable. No filler words...") before asking for a score. This rubric-conditioned prompting is what separates Tenacious-Bench tone scoring from a generic "rate this email 1–5" instruction.

Prometheus 2's finding that a small fine-tuned judge (7B) can match a 70B generic model on structured evaluation tasks supports the Path B architecture choice: a small Qwen3-0.6B or 1.7B model, fine-tuned on 62 Tenacious preference pairs, can outperform a larger generic model on Tenacious-specific decision correctness. The key is domain-specific training, not model scale.

---

## Where I Disagree

Prometheus 2's feedback collection is domain-agnostic: it covers reasoning tasks, coding tasks, and open-ended generation across a broad distribution. This works for general-purpose judge training because the evaluation criteria (coherence, helpfulness, safety) transfer across domains.

**Tenacious-Bench evaluates against domain-specific policy constraints that cannot transfer from a general feedback collection.** The dual-control rubric requires knowing that `confidence ≥ 0.65 + reply_intent == POSITIVE → autonomous action is correct`. The bench-capacity rubric requires knowing that `capacity_claim > bench_state.rust.available_engineers → violation`. These are specific to Tenacious's pricing sheet, policy document, and bench_summary.json. A Prometheus 2 model trained on GPT-4 feedback on general tasks has zero exposure to these constraints.

**My evidence**: This is the core argument for Path B over a Prometheus 2 adapter. If I fine-tuned Prometheus 2 on general feedback and then prompted it with a Tenacious-specific rubric, it would produce coherent-sounding feedback but would not reliably apply the capacity ceiling or dual-control policy constraints without seeing training examples of correct vs. incorrect applications. The preference pairs in training_data/preference_pairs.jsonl are explicitly constructed to teach this policy: rejected completions violate the policy, chosen completions satisfy it.

**Where Prometheus 2's architecture is directly applicable**: the feedback-explanation training objective. Kim et al. show that training the judge to produce a feedback explanation before the score improves calibration. For v0.2, Tenacious preference pairs should include a `chosen_rationale` field explaining *why* the chosen output satisfies the rubric. The current pairs have only (chosen, rejected) with no rationale — which is a gap relative to Prometheus 2's training data format.

---

## Implication for Tenacious-Bench

Prometheus 2 is not the right starting model for Tenacious-specific judging because its general training data cannot teach domain-specific policy constraints. Path B's design — training a small backbone on Tenacious-specific preference pairs with explicit policy violations in the rejected completions — is the correct approach. The one technique to borrow from Prometheus 2 is the feedback-explanation objective for v0.2 pair construction.
